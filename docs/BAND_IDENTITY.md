# 委託文: PR-F — バンドアイデンティティ基盤

発注日: 2026-07-19 ／ 基点: PR #24マージ後のmain ／ 1PR・Terra級
branch→Draft PR→発注者squash merge ／ 報告は日本語
正典: **同送の BAND_IDENTITY_v1.md（v1.1）**。色・様式の値はすべてこの表から転記し、本書と併せて `docs/BAND_IDENTITY.md` として収載すること。

## 作業

### 1. バンド色のデータ化
- `BAND_DATABASE` 全22組へ `color` フィールド新設（v1.1表のhexを転記）。`markStyle: 'A'|'B'|'C'` も同時に新設（様式割当表から転記）。
- **バンド文脈の色参照をDB直読へ移行**: メール差出人帯・SNSアバター・タグ帯等でバンドが送信者/主体の場合は `color` を使用。`bandColorIndexFromName()` のハッシュ振り分けは**非バンド差出人（店長・機材レンタル等）専用として存置**（全廃しない。バンド文脈での使用ゼロをgrepで証明）。

### 2. ワードマーク（22ロゴのCSS作字）
- 3様式をclassで実装: **A**=Anton＋skewX(-8deg)の擬似伊体・大文字（display級のため傾け可）／**B**=Barlow Condensed・大文字・letter-spacing .3em／**C**=Barlow大文字＋文字を横切るBG色2pxの欠きライン1本（::after）。
- 各ワードマークに `color` の3px帯（下端）を付帯。画像アセット・外部フォント追加は禁止（同梱Anton／Barlowのみ）。

### 3. キャラ画像の器（3段フォールバック機構）
- メンバー／NPCデータへ `portraitId` 新設（既存 `sil_` 命名を踏襲。NPCはシーン側の既存IDを正とし、メンバーは `member_xxx` から機械的に付与）。
- 参照解決: `assets/char/<portraitId>_story.webp`（3:4）／`<portraitId>_avatar.webp`（1:1）→ 存在すれば表示、無ければ**ステンシルシルエット**（storyは既存 `.story-silhouette` 形状を継続、avatarは同形状type別のミニ版を転用実装）→ それも不可ならイニシャル（現行）。
- 画像は現時点でゼロで良い。**ダミー画像1枚を仮置きして3段すべての経路を動作証明**し、検証後にダミーは削除（機構だけ残す）。以後は発注者がファイルを置くだけで反映される状態にすること。

### 4. 未実装3画面へのビジュアル追加
- **バンド図鑑**: 一覧・詳細にワードマーク＋色帯＋代表キャラ名を表示（22組が一望できる画面＝色・様式の実機確認台を兼ねる）。
- **ライブ結果**: 対バン・イベント相手が存在する場合、結果ヘッダへ相手のワードマーク小＋色帯。fix4のコレオタイムラインは不変（表示要素の追加のみ）。
- **メール詳細**: 差出人がバンドの場合、ヘッダへワードマーク小＋色帯。

## 禁止事項

凍結事項・計算式・テキスト内容の変更（文体レーンの領分）。FESラインナップ序列・フライヤー進化（次段PR）。デッドコード3群（RIVAL_BANDS／旧候補25名／renderTopStats死定義）への接触。レイアウト再配置。

## 受け入れ基準

バンド文脈でのハッシュ色参照ゼロ（grep証跡）／図鑑での22組ワードマーク＋色の全景スクショ（`docs/pr-f-evidence/` 収載）／3段フォールバックの動作証跡（画像あり・ステンシル・イニシャルの3経路）／実機確認5項目の対比スクショ素材（朱vsACC・レモンvsFAME金・フレッシュグリーンvsMONEY・ワインの黒地視認・スチールvsDIM——**最終判定は発注者の実機目視**）／320-430px×全画面監査＋縦到達性監査PASS／両回帰PASS／game.js差分の分類表（データ追加・表示系のみ）／罠手順証跡（図鑑・結果・メール詳細のrender系はwrap-by-reassignment標準＋実描画確認）。
